  <action>
    Replace the Plan 01 logging.rs with a full implementation that:
    1. Initializes tracing-subscriber with env-filter for terminal (stdout)
    2. Configures file output via tracing-appender (or a simple file writer)
    3. Uses platform-appropriate log directories per D-14

    Since adding `tracing-appender` as a dependency adds complexity, implement the file logging through `tauri-plugin-log`'s file target (which already handles platform-log-dir resolution per D-14). The tracing-subscriber handles structured formatting for stdout (dev), while tauri-plugin-log handles file output.

    Implementation:

    ```rust
    use tracing_subscriber::{fmt, prelude::*, EnvFilter};
    use tracing_subscriber::layer::SubscriberExt;

    /// Initialize tracing for structured logging.
    /// 
    /// Architecture (two-tier per STACK.md):
    /// 1. tracing-subscriber (fmt + env-filter) → stdout/stderr for dev
    /// 2. tauri-plugin-log → platform log dir file output
    /// 
    /// File paths per D-14:
    /// - macOS: ~/Library/Logs/com.vivipet.app/vivipet.log
    /// - Linux: ~/.local/share/com.vivipet.app/logs/vivipet.log  
    /// - Windows: %APPDATA%/com.vivipet.app/logs/vivipet.log
    /// 
    /// Note: tauri-plugin-log's TargetKind::LogDir handles the
    /// platform-specific path resolution automatically.
    pub fn init() {
        // Tracing layer for stdout (dev mode, formatted output)
        let fmt_layer = fmt::layer()
            .with_target(true)
            .with_thread_ids(true)
            .with_file(true)
            .with_line_number(true);

        // Env filter: default "info", override via RUST_LOG env var
        let filter_layer = EnvFilter::try_from_default_env()
            .unwrap_or_else(|_| EnvFilter::new("info"));

        // Combine layers (tracing-subscriber registry)
        tracing_subscriber::registry()
            .with(filter_layer)
            .with(fmt_layer)
            .init();

        // Log init confirmation via tracing
        tracing::info!("Tracing initialized (stdout + file via tauri-plugin-log)");
        tracing::debug!("Debug logging enabled");
    }
    ```

    **Critical design decisions:**

    1. **File output is handled by tauri-plugin-log, NOT by tracing-appender.**
       Rationale: `tauri-plugin-log` already implements platform-appropriate log file paths (D-14). Adding `tracing-appender` would duplicate this functionality. The plugin's `TargetKind::LogDir` writes to the correct OS-specific directory using the app's bundle identifier (`com.vivipet.app`). This is configured in Plan 01's lib.rs plugin registration:
       ```rust
       .plugin(tauri_plugin_log::Builder::new()
           .level(log::LevelFilter::Info)
           .target(tauri_plugin_log::Target::new(
               tauri_plugin_log::TargetKind::LogDir {
                   file_name: Some("vivipet.log".into()),
               },
           ))
           .build())
       ```
       
       If Plan 01's lib.rs does NOT yet configure the log file target (it may only use default settings), update lib.rs to add the `Target::TargetKind::LogDir` configuration in this task. Check the current lib.rs content:
       ```bash
       grep -n "tauri_plugin_log" apps/desktop/src-tauri/src/lib.rs
       ```
       If `tauri_plugin_log::Builder` already exists but without file target, modify it to add the LogDir target. If it doesn't exist, add it to the plugin chain.

    2. **tracing.rs is separate from tauri-plugin-log configuration:**
       - `tracing::init()` handles the formatting layer for stdout
       - `tauri_plugin_log` handles the file output layer (configured in lib.rs)
       - Both work together: tracing macros (`info!`, `error!`) flow through tracing-subscriber (stdout) and simultaneously through `log` crate bridge → tauri-plugin-log (file + WebView console)
       
       This two-tier architecture matches STACK.md recommendation.

    3. **Add a log test on startup:** In lib.rs setup, add a few log messages at different levels to verify both stdout and file output work:
       ```rust
       tracing::info!("ViviPet Tauri app starting");
       tracing::warn!("This is a warning log message (Phase 1 test)");
       tracing::error!("This is an error log message (Phase 1 test) — not an actual error");
       ```

    4. **If lib.rs already has tracing::info! calls from Plan 02** (e.g., in window and tray modules), they will automatically flow through both stdout and file output. No changes needed to those modules.

    File output verification approach:
    - After running `cargo tauri dev` and the app starts, check the log file:
      ```bash
      # macOS
      cat ~/Library/Logs/com.vivipet.app/vivipet.log
      # Linux  
      cat ~/.local/share/com.vivipet.app/logs/vivipet.log
      # Windows (PowerShell)
      type "$env:APPDATA\com.vivipet.app\logs\vivipet.log"
      ```
    - The file should contain timestamped log entries with `[INFO]` level messages from app startup.
    - If the log file doesn't exist or is empty after app start, debug by checking:
      - Is `tauri_plugin_log::Builder` configured correctly?
      - Is the bundle identifier `com.vivipet.app` matching the log directory path?
      - Are there any Rust panics during plugin initialization?

    // Tauri IPC bootstrap test — verify frontend can call Rust commands
    async function testTauriIPC(): Promise<void> {
      try {
        const { invoke } = await import('@tauri-apps/api/core');
        const { getName } = await import('@tauri-apps/api/app');
        
        // Test 1: greet command (defined in commands/mod.rs)

        console.log('[Tauri IPC] All bootstrap tests passed');
      } catch (err) {
        console.warn('[Tauri IPC] Bootstrap test failed (expected if not in Tauri):', err);
      }
    }
